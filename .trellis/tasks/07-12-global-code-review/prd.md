# 全局代码审查

## Goal

对 ESP-IDF 固件及其本地 Web 界面进行全局代码审查，并修复已确认的跨任务运行时竞态和 Web BMS 扫描结果未自动刷新问题。

## Confirmed Facts

- 仓库为单仓库 ESP-IDF 工程，核心模块包括运行时、BMS/LVGL 桥接与 UI、FarDriver 协议、音频反馈及 `main/web` 本地 Web 界面。
- 当前位于 `main` 分支；用户已授权修复审查中确认的问题。
- 项目约束要求：修改符号前必须执行 GitNexus 影响分析；审查报告需要覆盖受影响的执行流与测试缺口。

## Requirements

- 对当前已提交基线中的应用源代码、组件接口、构建配置及本地 Web UI 进行静态审查。
- 检查正确性、错误处理、内存/资源生命周期、FreeRTOS 并发与 LVGL 线程安全、网络/协议边界、配置一致性以及测试覆盖。
- 每项发现必须包含严重级别、文件与行号、可验证的证据、影响说明与可行的修复方向；不以纯风格偏好作为缺陷报告。
- 结合 GitNexus 的执行流和影响面信息，说明关键运行时模块的回归风险。
- 使用一个覆盖 runtime 状态与 snapshot 的递归锁，避免 NimBLE 回调、主循环和 HTTP 处理器并发读写；锁外只消费一致的副本。
- Web BMS 扫描应显示进行状态，并在有限时间内轮询候选列表直到扫描结束。

## Acceptance Criteria

- [ ] 输出按严重程度排序的审查结果；若未发现阻断问题，明确说明审查范围和剩余风险。
- [ ] 每个问题都可定位到具体代码位置，并说明触发条件或违反的约束。
- [ ] 列出已运行的构建、静态检查或自测，以及未能覆盖的区域。
- [ ] 不修改产品源代码、用户未提交改动或 Git 历史。
- [ ] BMS 扫描请求后，Web UI 能在扫描完成前自动刷新候选列表，并在超时后恢复可操作状态。
- [ ] ESP-IDF 构建与 FarDriver 自测通过；运行时锁覆盖主循环、NimBLE 回调和 HTTP snapshot 读取路径。

## Out of Scope

- 无关的重构、刷写硬件。

# 全局代码审查执行计划

1. 记录提交基线、源码清单、组件依赖和已有测试。
2. 使用 GitNexus 读取仓库上下文、执行流和当前基线变化映射；对关键运行时符号执行上游影响分析。
3. 分层审查启动与硬件初始化、运行时 Wi-Fi/HTTP/BLE/NVS、LVGL 桥接/UI、FarDriver 协议、音频与 Web 页面。
4. 执行可用构建及 FarDriver 协议自测；记录命令及结果。
5. 复核每项发现的代码位置、触发条件和调用链，剔除无法证实的意见。
6. 产出按严重程度排序的审查报告及残余风险说明。
7. 为 runtime 增加递归锁和一致 snapshot 复制接口；在主循环、NimBLE/事件回调以及 HTTP 读取路径接入。
8. 为 `/api/bms/candidates` 暴露扫描状态；Web UI 在发起扫描后按状态轮询、超时收尾。

## 审查闸门

- 不修改产品代码；需要修复时另建或切换到修复任务。
- 高/严重问题必须包含明确调用路径或确定的 API/内存约束依据。
- 因硬件环境无法验证的结论必须标识为静态风险，不能宣称已复现。
- 互斥锁不能包住 HTTP 发送或 LVGL 绘制；任何新增锁必须在所有错误返回路径释放。

## 验证命令

- `./scripts/esp-idf-env.sh build`
- 按测试构建配置运行 `tests/fardriver_protocol_selftest.c`（若当前工具链支持）
- GitNexus 执行流、影响面和变更检测

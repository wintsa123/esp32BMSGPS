# 全局代码审查设计

## 范围与边界

- 审查对象为 `main/`、`components/`、`tests/`、构建/分区配置以及嵌入式和 Vercel Web UI。
- 审查以当前 `main` 提交基线为准；Trellis 任务与项目说明文件的未提交变更不纳入产品缺陷判断。
- 审查不修改产品代码、不刷写设备、不提交代码。

## 审查方法

1. 读取构建配置和组件依赖，建立启动、显示/触摸、运行时网络/BLE、协议和 Web API 的调用边界。
2. 使用 GitNexus 查询关键执行流，并在分析关键符号前记录上游影响面。
3. 运行时修复使用递归 FreeRTOS mutex：主循环入口、NimBLE 回调和 HTTP snapshot/API 读取均在锁内访问共享 runtime；LVGL 更新和 HTTP 响应只使用锁内获取的一致副本，避免长时间持锁。
4. 检查 HTML/JavaScript 的 API 认证、状态转换、错误处理及与后端契约的一致性。
5. 运行可用的主机构建和协议自测；将硬件依赖验证明确标为未覆盖。

## 报告契约

- 仅报告可证实的缺陷：`严重/高/中/低`，附 `文件:行号`、触发条件、影响、证据与修复建议。
- 结论按严重程度排序；风格建议与未证实猜测不计入缺陷。
- 单列已验证项、未覆盖项及总体合入建议。

## 风险控制

- 硬件、真实 BMS 和射频行为不能由静态审查完全验证，报告中保留该残余风险。
- 对高影响调用链优先通过 GitNexus 上游影响分析和执行流复核，避免孤立判断。
- 不在 mutex 持有期间等待 HTTP 发送或执行 UI 绘制；NimBLE 回调仅持锁完成状态转换并立即释放。

# Android 低延迟遥控投屏

## 目标与用户价值

在无 PSRAM 的 ESP32-WROOM-32E 仪表上，让 Android 10+ 手机在连接设备 Setup AP 后，以低延迟、低帧率镜像手机画面，并把设备触摸操作受限地回传到手机。用户无需安装或配置 Cast/Miracast 接收器。

## 已确认事实与约束

- 硬件为无 PSRAM、4 MB Flash 的 ESP32-WROOM-32E；全尺寸 240×320 RGB565 帧为 153,600 B，不能缓存完整帧。
- 当前固件已有 ESP-IDF HTTP 服务、受密码保护的 Setup AP（`192.168.4.1`）、LVGL 9.5 及 XPT2046 触摸。
- Setup AP 的 SSID/8 位数字密码策略和现有凭据二维码必须保持不变；AP 仍由设置页手动开关。
- 默认 UI 文案为中文；TFT 无中文点阵字体时只能显示 ASCII 标记或英文 ASCII 文案。
- Android 客户端为离线原生 Kotlin App，最低 Android 10；首版只交付本地可安装 APK，不上架 Google Play。

## 范围内需求

1. 固件提供认证的 `GET /api/cast/info` 和单客户端 `GET /cast` WebSocket，认证复用 Setup AP 密码。
2. 定义版本化二进制协议：帧开始、固定最大 16×16 RGB565 块、帧结束/ACK、心跳和输入事件；服务端须拒绝认证失败、第二客户端、版本不匹配、坐标/尺寸/负载越界的包。
3. 显示桥提供持 LVGL 锁的区域 RGB565 直写能力；投屏活动期间压制常规仪表刷新，结束时恢复仪表。
4. 主界面增加可手动横向切换到的第 4 页“远程投屏”。空闲页显示 ASCII 说明和含 SSID、密码、`192.168.4.1`、协议版本的深链二维码。
5. 投屏期间：首次触摸展示顶部导航；导航含退出入口。客户端断开、心跳超时、App 主动结束、设备退出均恢复仪表。
6. Android App 经系统确认连接 Setup AP 后，由用户主动点击开始投屏才请求 `MediaProjection`。按屏幕方向等比缩放并黑边填充；首帧完整块、后续差分块；最多一个 ACK 未确认帧，网络慢时丢旧采集帧。
7. App 接收设备单指点击、长按、拖动、滑动事件，提供 Back/Home；系统级反控仅在用户自行启用无障碍服务后尝试，明确不支持文字输入、多点、DRM/安全内容和自动化。

## 不在范围内

- Miracast、Google Cast、Wi-Fi Direct、H.264/RTSP 接收器。
- 全帧缓冲、JPEG/H.264 解码，或对无 PSRAM 机型承诺高帧率。
- iOS 客户端、Google Play 发布、自动持久化热点密码。

## 验收标准

- [ ] `cast-info` 在正确认证时返回协议版本、逻辑分辨率、旋转、最大块大小、投屏状态；错误认证不泄露该信息。
- [ ] WebSocket 同时仅允许一个授权客户端，所有恶意/越界/过大包被拒绝且不造成分配随画面增长。
- [ ] 16×16 或更小 RGB565 块可直写；投屏结束始终恢复仪表页。
- [ ] 投屏页、深链二维码和现有 AP 凭据 UI 均满足项目语言与 SSID/密码约束。
- [ ] Android debug APK 可构建；单测覆盖缩放/旋转/差分和协议编码。
- [ ] 固件可构建，尺寸与 heap 审计无完整帧分配；在真机上验证与 BMS、BLE、热点并行时没有 watchdog/panic，交互约 1–3 fps。
- [ ] 验证断线、超时、权限撤销、顶部退出、App 退出、热点未开提示与二维码唤起；新增预览图在 `preview/`。

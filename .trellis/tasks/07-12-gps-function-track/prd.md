# GPS 基础接入：速度、启动计时与 PPS 日志

## 目标

可靠接入 336H GPS。轮播页中的 GPS 页面仅显示实时 GPS 速度与设备本次启动后的计时；GPS NMEA 时间和 GPIO35 PPS 授时状态仅写入串口日志，供硬件联调验证。

## 已确认事实

- 硬件为 ESP32-WROOM-32E（4 MB Flash、无 PSRAM）；336H GPS 通过 UART NMEA 工作，实机原始字节验证其当前波特率为 115200（并非模块资料所述的出厂默认 9600）。
- 调试阶段 NMEA 接线为 GPS TX → ESP32 UART1 RX/GPIO27，GPS RX ← ESP32 UART1 TX/GPIO18；GPIO18 后续启用 TF 卡时需归还 SPI SCK。当前运行时已经验证/解析 RMC 的定位有效标志和速度。
- PPS 接到 ESP32-WROOM-32E IO35，即 ESP32 GPIO35（RTC GPIO5 / ADC1_CH7）；该脚仅输入且无内部上下拉，输入必须为不超过 3.3 V 的单路信号。
- UI 已有轮播 GPS 页面容器和全局 `GPS OK/GPS --` 状态，但 GPS 页面尚无内容。运行时当前只在 GPS 页面激活时轮询 GPS UART，离开该页面会清空 UART 接收数据。
- 主循环每 50 ms 调用一次运行时 tick，运行时已有 `tick_count` 秒计数；UI 数据通过 dashboard snapshot 更新。

## 本期范围

- 连续接收、校验并解析 RMC，保持 GPS 有效定位和实时速度状态；离开 GPS 页面也不能停止接收。
- 为 GPIO35 配置 PPS 上升沿中断，安全记录 PPS 事件；PPS 缺失或异常不能阻断 NMEA 速度显示。
- GPS 页面显示实时速度（沿用已有 km/h/mph 设置）与本次启动时长；无有效定位时显示不可用状态而不沿用旧速度。
- 在串口日志输出 GPS 授时联调信息：有效 RMC UTC 时间/日期、PPS 首次接收、PPS 丢失/恢复以及与系统秒计数的关系。日志应限频，不能每秒淹没串口或干扰 UART0 NMEA。

## 明确不包含

- TFT/Web 的 GPS 日期/时间显示、行程平均速度、最高速度、行驶时间。
- TF 卡检测、挂载、空间检查、自动保存、历史读取，以及所有 Web GPS API。
- 轨迹点、轨迹线、地图、离线地图、GPX/KML 导出。

## 约束

- UART0 同时用于烧录/日志与 GPS，所有新增日志必须简短、限频，并不得在 UART 接收路径或 GPIO ISR 中打印。
- GPIO35 无内部上下拉；实现不应启用不存在的内部上下拉。ISR 只更新原子/volatile 状态，格式化和日志放在主循环完成。
- 所有界面默认中文；本期 TFT 仅使用现有 ASCII 数字/单位，避免引入字库变更。

## 验收标准

- [ ] GPS 页和其他轮播页间切换后，GPS 仍持续解析；重新进入 GPS 页时显示最新有效速度。
- [ ] GPS 有效定位时，GPS 页显示实时速度与单位；无有效定位时显示明确不可用状态。
- [ ] GPS 页显示从本次启动开始、单调递增的时长，格式清晰，且不受 GPS 是否定位影响。
- [ ] GPIO35 PPS 被配置为输入中断；串口日志可看到限频的 PPS 接收/丢失/恢复及有效 RMC UTC 授时信息。
- [ ] PPS 未接、NMEA 未接、校验失败或 GPS 页未激活时，主循环、BMS、控制器、触摸、Wi-Fi 和 UI 均保持可用。

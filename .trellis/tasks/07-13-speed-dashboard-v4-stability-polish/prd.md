# 速度仪表 V4 稳定性与绘制优化

## Goal

修复速度仪表 V4 的三项真机体验问题：GPS 静止速度漂移并补足定位状态失效诊断、轮播切页卡顿、弯曲速度色带出现三角形内部接缝。在 ESP32-WROOM-32E（4 MB Flash、无 PSRAM）上保持现有统一速度来源、横竖屏布局和数据语义不回退。

## Background

- 用户真机静止放置时，GPS 速度显示在 `1–3 km/h` 间波动；继续放置后 GPS 状态点由绿色变为橙色。
- GPS 状态点当前语义为：绿色表示定位有效，橙色表示定位无效。已有真机日志曾在室外看到约 5 颗卫星、至少一颗 `SNR≈25`，约 84 秒首次 PPS，120–180 秒内 `fix=1`、`pps=1`；因此必须区分静止漂移、RMC 状态无效、RMC 超时、PPS 丢失及 UART/NMEA 解析异常。
- 已有真机堆日志显示初始化约 200 KB 空闲、首屏完成约 111 KB、最大连续块约 106 KB，未见 panic/WDT。轮播卡顿目前更像速度仪表高频全量重绘造成的耗时，仍需用测量确认，不能只按“内存爆掉”处理。
- V4 色带约由 32 段四边形组成，每段拆成两个抗锯齿三角形；当前可见的内部细线疑似三角形抗锯齿覆盖接缝。
- 当前横屏弧线并未调用 `lv_bezier3()`；`speed_dashboard_smooth_step()` 使用的 `3t^2-2t^3` 与控制值 `0,0,1024,1024` 的三次贝塞尔等价。320×240 下的主要平滑度上限来自 32 段折线采样（横屏每段约 9 px），不是 10 位定点精度。
- 目标设备为 ESP32-WROOM-32E、ST7789 320×240、ESP-IDF 5.5.4、LVGL 9.5，无 PSRAM。
- 当前代码没有 GPS 低速死区或迟滞：RMC 速度由 `runtime_apply_gps_line()` 原样写入 `gps_speed_knots_milli`，再由 `runtime_update_snapshot_speed()` 换算为显示单位（`components/esp_bms_idf_runtime/esp_bms_idf_runtime.c:4865`、`:1348`）。行程开始后，平均电耗距离也会积分同一原始低速，因此静止漂移不仅影响主数字，还可能产生幽灵距离（`components/esp_bms_idf_runtime/esp_bms_speed_dashboard.c:92`）。
- 定位状态由 RMC 字段 2 的 `A/V` 决定；连续 3 秒未收到可解析 RMC 时也会强制清除定位并变橙。PPS 丢失只记录状态，不直接改变颜色（`components/esp_bms_idf_runtime/esp_bms_idf_runtime.c:1233`、`:4999`）。
- 2026-07-13 只读真机监控冷启动捕获到 `$GNRMC,...,V,...`，60 秒汇总仍为 `fix=0 pps=0`；同时捕获到残句 `,,,145030.00,V,N*57` 被报告为 `invalid RMC`。当前 96 字节行缓冲溢出后会立即从零继续收集尾部，而不是丢弃到下一条 `$`/换行，四频多星座模块的长 GSA/GSV 输出会暴露该分帧缺陷（`components/esp_bms_idf_runtime/esp_bms_idf_runtime.c:4935`）。
- `set_gps_dashboard()` 在每次 snapshot 应用时无条件让整张 `speed_art` 失效；主循环至少因每秒 uptime 变化而持续应用 snapshot（`components/esp_bms_lvgl_ui/esp_bms_lvgl_ui.c:6407`、`main/idf_main.c:153`）。
- 页面滚动每帧启用了整屏 viewport 失效；该选项最初用于避免局部刷新造成页面分块/残影，不能在没有 A/B 真机验证时直接关闭（`components/esp_bms_lvgl_ui/esp_bms_lvgl_ui.c:6717`、`:7115`，`sdkconfig.defaults:123`）。
- LVGL 软件三角形每次绘制都会动态申请并释放一行 mask 缓冲；当前一帧色带至少绘制 64 个三角形，再叠加轮廓、刻度、卫星和电池，足以造成滚动帧 CPU/分配压力，但不等同于内存泄漏（`managed_components/lvgl__lvgl/src/draw/sw/lv_draw_sw_triangle.c:47`）。

## Requirements

### R1. GPS 静止速度与定位状态

- 仅对实际来源为 GPS 的速度应用静止抑制，不改变控制器速度语义。
- GPS 规范化速度采用 `2/4 km/h` 迟滞：静止态原始速度低于 `4 km/h` 时保持 `0`，达到 `4 km/h` 后进入运动态；运动态降到 `2 km/h` 及以下回到 `0`。公英制只转换显示结果，迟滞阈值始终以 GPS 公制基准计算。
- 主速度和本次开机行程距离积分共用规范化 GPS 速度，避免停车后低速漂移产生幽灵里程；控制器速度和 GPS 定位有效状态不受该滤波影响。
- 不得用速度归零掩盖定位失效；GPS 状态点仍必须反映真实定位有效性。
- 查明状态点变橙的直接触发路径，并为 RMC 无效、数据超时、PPS 状态和 NMEA 解析提供足够的可验证依据。
- 保留现有 GPS 时间、速度来源偏好/实际来源和控制器离线回退语义。

### R2. 轮播切页性能

- 定位轮播卡顿的主因，并用切页耗时、刷新频率及堆/最大连续块数据验证结论。
- 速度仪表只在会影响绘制结果的数据或布局变化时重绘；非可视页不应产生不必要的高成本色带重绘。
- 优化后电池页 → 速度仪表页 → 投屏页的手动滑动和程序化动画切换均应流畅，不引入触摸、旋转、设置导航或数据更新回归。
- 优化不得依赖 PSRAM，也不得以降低实时数据正确性为代价。

### R3. 速度色带精细度

- 消除或在真机可视条件下不可察觉地降低每段内部及相邻段之间的三角形接缝。
- 横屏弧线不得出现肉眼可见的折线转折；优先把现有等价贝塞尔曲线从 32 段提高到 48 段，并按段数比例保持危险区起点、主刻度和次刻度位置不变。
- 不为公式等价的曲线改写引入 `lv_bezier3()`、浮点运算、查表或后台计算任务；仅在 48 段预览或真机仍不足时再提高采样密度。
- 保留 V4 的弯曲分段外形、已行进区渐变、未行进区深灰、高速危险区红色、白色外轮廓和超量程封顶语义。
- 横屏 320×240 与竖屏 240×320 共用同一套绘制逻辑，均不得出现明显裂缝、越界或闪烁。

### R4. 兼容与范围

- 保留上一任务已完成的状态栏收窄、中文温度标签、小电池/小挡位、无控制器信号默认 1 挡以及无 BMS 时隐藏电池和电耗。
- 保留现有未提交或并行任务改动；所有修改采用增量合并，不覆盖用户工作。
- 新增或更新的 LVGL 预览图片必须放在仓库根目录 `preview/`。
- TF 卡、历史骑行、Web 历史查询不在本任务范围内。

### R5. 桌面 LVGL 模拟器

- 桌面端必须直接编译并运行 `components/esp_bms_lvgl_ui/esp_bms_lvgl_ui.c` 和同一组固件字体，不再用 Python 重写 UI 作为主要验收依据。
- 使用仓库内 LVGL 9.5.0 和系统已有 SDL2；主机入口只替换显示/输入驱动及少量 ESP-IDF API，不复制页面、控件、事件或自绘逻辑。
- 模拟器至少支持 320×240 横屏、240×320 竖屏、鼠标/触摸等价操作，以及速度、定位、BMS、控制器和单位等快照状态切换。
- 提供可重复的构建/运行命令和无显示环境冒烟测试；模拟器不得成为固件构建输入，也不得改变设备运行时契约。

## Acceptance Criteria

- [ ] GPS 为实际速度来源且设备静止时，连续观察主速度不再在 `1–3 km/h` 波动；正常起步可退出静止状态，减速停车可稳定回到 `0`。
- [ ] 验证 `3.9→4.0 km/h` 进入运动态、`2.1→2.0 km/h` 回到静止态，阈值附近往返不抖动；mph 显示与行程距离使用同一规范化结果。
- [ ] GPS 状态点变橙时，日志或快照能明确对应定位无效/超时等真实状态；不能因显示滤波伪造绿色状态。
- [ ] 验证 RMC 有效与无效、RMC 超时、PPS 存在与丢失，以及包含 GSA/GSV 的连续 NMEA 输入不会污染 RMC 解析。
- [ ] 记录优化前后的仪表刷新次数或耗时、轮播切页耗时、空闲堆和最大连续块；优化后切页无肉眼可见停顿，且无持续内存下降、panic 或 WDT。
- [ ] 速度色带在横竖屏、无效速度、0、低速、中间值、满量程及超量程预览/真机下无明显三角形内部接缝。
- [ ] 横屏 320×240 预览和真机上，弧形外轮廓及色带中心线无明显 32 段折线感；危险区、刻度和速度映射位置与优化前一致。
- [ ] GPS/控制器来源切换、控制器断连/重连、km/h↔mph、横竖屏旋转、无 BMS 隐藏电池与电耗均保持正确。
- [ ] 桌面 SDL 模拟器运行真实 `esp_bms_lvgl_ui.c`，横竖屏均可启动，鼠标可操作页面/设置，主机快捷键可更新关键快照状态，且无头模式可完成启动和更新冒烟测试。
- [ ] 通过相关主机测试、字体/预览检查、`git diff --check`、ESP-IDF 构建、GitNexus `detect_changes()` 和 LAN RFC2217 真机刷写监控。

## Notes

- 本任务属于跨 GPS 解析、运行时快照、LVGL 绘制和真机验证的复杂任务；技术边界和执行门槛分别记录在同目录 `design.md` 与 `implement.md`，开始实现前须由用户审核批准。
